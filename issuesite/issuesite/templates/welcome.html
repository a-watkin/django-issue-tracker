Welcome

So I've tried to meet all your use cases. I wasn't really sure what role the staff account described in the specifications should have. Whether you wanted a longin for account or something along those lines.

Since it's not mentioned explicitly and the staff role described can't do anything I've implemented the issues list without any login.

Instead I've assumed that staff users can solve problems but only admin can accept solutions and attribute them to that staff user. So staff users are simply the solver in the database model.

The admin account can create, change and delete issues. It can also do the same for StaffUsers.

Admin account login:

  • username: admin
  • password: adminpassword

Staff role is implemented as: StaffUser in the model. One StaffUser has been created via the admin console. StaffUser credentials:

  • username: a_staff_user
  • password: staffpassword
The StaffUser account cannot login or alter anything, it can only be assigned as the solver to an issue.
